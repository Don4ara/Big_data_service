# Data Vitrine Mock API

Это backend-приложение на базе **NestJS**, которое служит песочницей/генератором фейковых данных (заказов из ресторанов). Оно генерирует JSON-объекты со случайными, но правдоподобными данными для тестирования витрин данных, аналитики или других сервисов.

## Как развернуть и запустить у себя

### Требования
- Node.js (рекомендуется v18 или v20+)
- npm или yarn

### Шаги установки:
1. Склонируйте репозиторий и перейдите в папку проекта:
   ```bash
   git clone <url-вашего-репозитория>
   cd Big_data_service
   ```
2. Установите зависимости:
   ```bash
   npm install
   ```
3. Запустите проект в режиме разработки (watch mode):
   ```bash
   npm run start:dev
   ```
Приложение запустится локально. По умолчанию API будет доступно по адресу `http://localhost:3006`.

## 📡 Доступные эндпоинты (API)

Все методы находятся по пути `/data-vitrine`.

### 1. `GET /data-vitrine/generate`
Генерирует новые заказы, возвращает их списком (в формате JSON) и сохраняет в оперативной памяти (временно).
- **Параметры:** `?count=10` (необязательный, по умолчанию 10, максимум 5000)
- **Пример вызова:** `http://localhost:3000/data-vitrine/generate?count=5`

### 2. `GET /data-vitrine/stream`
Бесконечный поток данных (Server-Sent Events - SSE).
Каждую секунду автоматически генерирует от 1 до 5 новых заказов и отправляет их клиенту. Идеально для тестирования реал-тайм дашбордов.
- **Пример:** можно подключиться через `EventSource` в браузере или утилитами, поддерживающими SSE.

### 3. `GET /data-vitrine/orders`
Возвращает абсолютно **все** заказы, которые были сгенерированы или добавлены за время текущей работы сервера (хранятся в памяти).

### 4. `POST /data-vitrine/orders`
Позволяет вручную добавить свой собственный объект заказа в общее хранилище в памяти.
- **Тело запроса:** Структура заказа в формате JSON.
- Недостающие поля, такие как `orderId`, `createdAt`, `updatedAt`, будут сгенерированы автоматически, если вы их не передадите.

## 🏗 Структура JSON-файла заказа и форматы данных

Каждый сгенерированный заказ имеет глубокую и разнообразную структуру. Ниже приведено описание полей и возможных форматов.

### Пример корневой структуры
```json
{
  "orderId": "874593021",
  "orderDate": "2024-02-15T12:00:00.000Z",
  "currency": "RUB",
  "customer": { ... },
  "restaurant": { ... },
  "orderContent": { ... },
  "courier": { ... },
  "financialSummary": { ... },
  "status": "Доставлен",
  "review": { ... },
  "createdAt": "2024-02-15T11:00:00.000Z",
  "updatedAt": "2024-02-15T12:05:00.000Z"
}
```

### Разнообразие форматов (Специально для тестирования)

Система намеренно генерирует данные в различных форматах (грязные данные), чтобы имитировать реальные неконсистентные источники:

**1. Древо дат заказа (`orderDate`, `createdAt`, `updatedAt`)**
- `orderDate`: Дата создания заказа. Может приходить в трех форматах:
  - ISO 8601 строка: `"2024-02-15T13:45:30.000Z"`
  - Unix Timestamp (число в секундах): `1708004730`
  - Локальная дата строкой (РФ): `"15.02.2024"`
- `createdAt`: Время сохранения записи в БД (рассчитывается как `orderDate` + 1-5 секунд).
- `updatedAt`: Время последнего изменения статуса заказа (рассчитывается как `orderDate` + случайный интервал от 0 до 5 часов).

**2. Телефон клиента (`customer.phone`)**
- Полный международный (с дефисами): `"+7-999-123-45-67"`
- Без форматирования (через 8): `"89991234567"`
- Без форматирования (через +7): `"+79991234567"`

**3. Номера телефонов курьера (`courier.phone`)**
- Строго в формате: `"+79991234567"`

**4. Часовые пояса (`deliveryTimeZone` / `timeZone`)**
- Смещение: `"+03:00"`, `"+05:00"`, `"+07:00"`
- Название зоны: `"Europe/Moscow"`, `"Asia/Yekaterinburg"`

### Описание вложенных объектов

* **customer (Покупатель)**
  * `customerId` — ID покупателя (числовая строка, 7 символов).
  * `fullName` — Имя и фамилия (Генерируется с вероятностью: 80% берется из нашего подготовленного словаря российских имен для большей реалистичности, 20% — полностью рандомное имя от Faker).
  * `phone` — Номер телефона.
  * `email` — Email пользователя.
  * `deliveryAddress` — Полный адрес: город, улица, номер дома с дробью (`15/2`), квартира, подъезд, этаж, домофон (с символом `#`) и почтовый индекс.
    * По сгенерированному адресу (городу) происходит **реальное геокодирование** (через API LocationIQ). Возвращаются координаты (`lat` и `lon`) и часовой пояс (`deliveryTimeZone`, например, `"Asia/Yekaterinburg"`).

* **restaurant (Ресторан)**
  * `restaurantId` (числовая строка, 4 символа), `brandName` (Бренд), `legalEntity` (Юр. лицо).
  * `address` — Полный физический адрес ресторана. В **98%** случаев город ресторана совпадает с городом покупателя. В **2% случаев** генерируется ошибка (несовпадение городов), при которой с вероятностью 80% статус заказа будет зависшим или "Отменен".
  * `taxInfo` — Информация о налогах: ИНН, КПП, `vatPercent` (20).
  * `workingHours` — Время работы (start: `"10:00:00"`, end: `"20:00:00"`, timeZone берется из геокодирования).

* **orderContent (Состав заказа)**
  * `items`: Массив позиций (блюд).
    * `productId` (числовая строка, 6 символов), `name` (Название генерируется случайно, 20% шанс что это десерт), `quantity` (от 1 до 7), `pricePerUnit`, `category` (Категория из словаря).
    * `specialInstructions`: Пожелания, например *"Меньше льда"*, *"Без лука, пожалуйста"* (с высокой вероятностью может быть `null`).
  * `options`: Настройки типа `numberOfCutlery` (количество приборов), `requiresContactlessDelivery` (бесконтактная доставка), `isEcoFriendlyPackaging` (эко-упаковка).

* **courier (Курьер)**
  * `courierId` (числовая строка, 5 символов).
  * `name` — Имя курьера (Генерируется по тому же правилу: 80% — из словаря курьеров (преимущественно популярные в доставке имена), 20% — Faker).
  * `transportType` — Способ передвижения: `"bicycle"`, `"car"`, `"walking"`, `"scooter"`.
  * `currentLocation` — Долгота (`lon`) и широта (`lat`).
  * `estimatedArrival` — Время прибытия строкой в разных случайных форматах: `"14:30"` (24-часовой), `"02:30 PM"` (12-часовой с AM/PM).

* **financialSummary (Финансы)**
  * `paymentMethod`: Способ оплаты (`"CARD_ONLINE"`, `"CASH"`, `"APPLE_PAY"`, `"GOOGLE_PAY"`, `"SBP"`).
  * `subtotal`, `taxAmount`, `deliveryFee` (может быть 0), `serviceFee` (может быть 0).
  * `discountAmount`: Различные варианты скидок или 0. Плюс плавающая скидка 10%.
  * `grandTotal`: Итоговая сумма заказа.

* **status & review (Статус и Отзывы)**
  * Статус может быть: `"Новый"`, `"Готовится"`, `"Передан курьеру"`, `"Доставляется"`, `"Доставлен"`, `"Отменен"`.
  * `review`: Отзыв появляется только у статуса `"Доставлен"` и только в **80%** случаев (`20%` — `review = null`).
    * **Рейтинг отзыва** считается из качества ресторана, задержки доставки, сложности заказа, случайного шума, бонусов за удачную доставку и штрафов за задержку/инциденты. Итоговый рейтинг всегда в диапазоне `0.5..5.0` и округляется шагом `0.5`.
    * **Текст отзыва** выбирается из позитивного словаря при рейтинге `>= 3`, иначе из негативного.

## Как считается рейтинг

Ниже описана логика, по которой сервис считает итоговую оценку отзыва.

### Когда отзыв вообще появляется

Отзыв создаётся только если:

- статус заказа = `"Доставлен"`
- дополнительно проходит шанс `80%`

Упрощённо это выглядит так:

```ts
if (status === 'Доставлен' && Math.random() > 0.2) { // отзыв есть только у части доставленных заказов
  const delayHours = hoursOffset + minutesOffset / 60; // итоговая задержка в часах
  const { rating } = buildReviewRating(...);          // считаем звёзды
  const comment = rating >= 3                         // если 3+ -> позитивный словарь
    ? randomChoice(positiveReviews)
    : randomChoice(negativeReviews);                  // иначе негативный словарь
}
```

### Из чего состоит вход в формулу рейтинга

В `buildReviewRating(...)` попадают:

- `quality` — текущее качество ресторана в этом сезоне рынка; оно не задаётся вручную, а собирается из базового профиля ресторана, сезонного/era-сдвига рынка, market bias и небольшого случайного jitter;
- `delayHours` — задержка доставки в часах;
- `itemsCount` — сколько позиций в заказе;
- `requiresContactlessDelivery` — была ли бесконтактная доставка;
- `isEcoFriendlyPackaging` — включена ли eco-friendly упаковка.

### Финальная формула

Итоговая оценка считается как сумма базовых факторов, бонусов и штрафов:

```ts
rating = roundToHalf(                        // шаг рейтинга = 0.5 звезды
  clampRating(                              // рейтинг зажимается в диапазон 0.5..5.0
    baseScore +                             // базовая сила заказа
    qualityLift +                           // плавный бонус за quality
    restaurantFactor +                      // вклад ресторана
    courierFactor +                         // вклад курьера
    complexityFactor +                      // влияние сложности заказа
    randomNoise +                           // небольшой случайный шум
    excellenceBonus +                       // бонус за очень хороший простой заказ
    perfectDeliveryBonus +                  // бонус за почти идеальную доставку
    premiumConsistencyBonus +               // бонус за стабильный premium-уровень
    legendaryServiceBonus -                 // редкий аплифт для топовых доставок
    delayPenalty -                          // штраф за задержку
    incidentPenalty                         // редкий штраф за инцидент
  )
);
```

Отсюда следует важное:

- рейтинг никогда не будет ниже `0.5`;
- рейтинг никогда не будет выше `5.0`;
- рейтинг округляется только в шаг `0.5`: `0.5`, `1.0`, `1.5`, `2.0` ... `5.0`.

### Что входит в каждый фактор

Сначала сервис считает само `quality` ресторана. Оно складывается из:

- базового профиля ресторана по `restaurantId` и стабильному `marketProfileSeed`;
- era/season momentum, чтобы ресторан не был навсегда прибит к одной точке;
- общего `marketQualityBias` сезона;
- маленького случайного отклонения внутри допустимых границ.

Упрощённый фрагмент кода выглядит так:

```ts
const baselines = getRestaurantBaselines(input.seasonKey, input.restaurantId);
const seasonShifts = getRestaurantSeasonShifts(
  input.seasonKey,
  input.restaurantId,
);
const qualityBounds = getQualityBounds(baselines.quality);

const targetQuality = clamp(
  applyBoundedMomentum(
    baselines.quality,
    seasonShifts.qualityMomentum,
    qualityBounds.upwardRoom,
    qualityBounds.downwardRoom,
  ) +
    input.marketQualityBias * 0.05 +
    input.randomFloat(-0.02, 0.02),
  qualityBounds.min,
  qualityBounds.max,
);

const quality = clampUnit(targetQuality);
```

Где:

- `baselines.quality` — базовый уровень ресторана;
- `seasonShifts.qualityMomentum` — сдвиг текущей эры/сезона;
- `input.marketQualityBias * 0.05` — общий сезонный bias рынка;
- `input.randomFloat(-0.02, 0.02)` — маленький случайный jitter;
- `qualityBounds` — границы, чтобы профиль не улетал слишком далеко вверх или вниз.

После этого уже определяется `quality band`:

| Band | Условие |
| --- | --- |
| `band 4` | `quality >= 0.90` |
| `band 3` | `quality >= 0.72` |
| `band 2` | `quality >= 0.45` |
| `band 1` | `quality >= 0.18` |
| `band 0` | `quality < 0.18` |

Именно band потом влияет на `restaurantFactor`, `courierFactor`, часть `complexityFactor`, `randomNoise`, шанс бонусов и шанс `incidentPenalty`.

| Фактор | Из чего берётся | Как считается | Значения / условия | Как влияет |
| --- | --- | --- | --- | --- |
| `baseScore` | `quality` | Если `quality >= 0.975`, база = `4.82 + randomChoice([0.03, 0.08, 0.12, 0.16])`; если `quality >= 0.95`, база = `4.68 + randomChoice([0, 0.05, 0.1, 0.14, 0.18])`; иначе база строится через `Math.pow(quality, 0.82)` и небольшой spread | premium-зона для топов, нелинейная кривая для остальных | Это главная стартовая база: сильный ресторан начинает заметно выше слабого |
| `qualityLift` | `quality` | Плавный бонус поверх базы: `lerp(0, 0.3, quality)` | От `0` до `+0.3` | Дополнительно тянет вверх хорошие профили |
| `restaurantFactor` | `quality band` | Для текущего band случайно выбирается значение из своего диапазона | `band 4`: примерно `+0.05..+0.34`; `band 3`: примерно `-0.10..+0.35`; `band 2`: от отрицательных до слегка положительных; `band 1/0`: чаще минус | Даёт живой разброс по “силе ресторана” |
| `courierFactor` | `quality band` | Считается похоже на `restaurantFactor`, но через отдельные диапазоны | У верхних band чаще нейтральный или положительный; у слабых чаще отрицательный | Моделирует качество исполнения доставки |
| `complexityFactor` | `itemsCount`, `requiresContactlessDelivery`, `isEcoFriendlyPackaging` | Сначала сервис определяет, сложный заказ или нет; потом выбирает поправку | Сложный заказ: `itemsCount >= 5` или есть contactless / eco; для простого заказа фактор чаще нейтральный или слегка положительный, для сложного — чаще отрицательный | Объясняет, почему даже хороший ресторан не всегда даёт `5.0` на тяжёлом заказе |
| `randomNoise` | Внутренний случайный шум | Выбирается маленькая случайная поправка из узкого диапазона | У топовых профилей шум чаще нейтральный/слегка положительный; у слабых — нейтральный/слегка отрицательный | Не даёт одинаковым заказам всегда одну и ту же оценку |
| `excellenceBonus` | `quality`, `delayHours`, простота заказа | Сначала проверяются условия, потом проходит случайный шанс и выбирается бонус | Только если заказ простой, `delayHours < 1`, `quality` уже хороший; значения: `0.25`, `0.35`, `0.5` | Первый бонус за очень удачную доставку |
| `perfectDeliveryBonus` | `quality`, `delayHours`, простота заказа | Считается так же, но с более жёсткими условиями и более сильным набором бонусов | Только если заказ простой, `delayHours < 0.45`, `quality >= 0.92`; значения: `0.25`, `0.35`, `0.5`, `0.65` | Уже заметно двигает заказ в верхнюю часть шкалы |
| `premiumConsistencyBonus` | `quality`, `delayHours`, сложность заказа | Размер зависит от того, насколько `quality` близок к 1, насколько маленькая задержка и насколько простой заказ | Только если `quality >= 0.90` и `delayHours < 1.1` | Дает устойчивую прибавку сильным ресторанам, а не только единичный всплеск |
| `legendaryServiceBonus` | `quality`, `delayHours`, простота заказа | Динамический верхний бонус для почти безупречных доставок | Только если `quality >= 0.955` и `delayHours < 0.75`; усиливается, если заказ простой и задержка почти нулевая | Именно этот фактор помогает доходить до `5.0` |
| `delayPenalty` | `delayHours` | Считается по ступенчатой шкале | `>= 1.0h -> 0.25`; `>= 1.5h -> 0.5`; `>= 2.5h -> 0.75`; `>= 3.0h -> 1.0`; `>= 4.0h -> 1.25`; `>= 4.5h -> 1.5`; `>= 5.5h -> 1.75`; `>= 6.0h -> 2.0`; `>= 7.0h -> 2.25`; `>= 7.5h -> 2.5` | Главный отрицательный фактор: длинная доставка быстро тянет рейтинг вниз |
| `incidentPenalty` | `quality`, редкое случайное событие | Сначала считается шанс инцидента от `quality`, потом выбирается величина минуса | Только если `delayHours < 3`; значения: `0.25`, `0.5`, `0.75`, `1.0`; у слабых профилей шанс выше | Даёт дополнительный минус за неприятный сбой поверх обычной задержки |

### Почему внутри есть `0.25`, а итог всё равно с шагом `0.5`

Дробные значения вроде `0.25`, `0.35` или `0.75` участвуют только внутри формулы как бонусы и штрафы. После расчёта итоговый score:

1. ограничивается диапазоном `0.5..5.0`;
2. округляется до шага `0.5`.

Поэтому в JSON всегда будут только `0.5`, `1.0`, `1.5` ... `5.0`, а медиана может быть `0.5`, если большая часть заказов после штрафов схлопывается в нижнюю границу.
